\chapter{System Overview}
This chapter presents the PawPal system architecture, data design, domain models, and detailed design diagrams. The system employs a modular monolithic backend, cross-platform mobile frontend, and specialized AI services.

\section{Architectural Design}
The system uses a six-tier architecture: presentation tier, API gateway tier, business logic tier, AI services tier, data storage tier, and external integrations tier. AI services run as isolated Python FastAPI microservices for efficient ML framework integration while all other backend logic is implemented as a Golang modular monolith.

\subsection{High-Level Architecture}
The PawPal high-level component architecture consists of a Flutter mobile client and a Next.js admin portal in the presentation layer, a Golang/Gin API gateway handling JWT authentication and routing, a modular monolith business logic layer with modules for User Management, Pet Profiles, Breed Identification, AI Chatbot, Marketplace, Community Hub, Veterinary Services, Caregiver Services, Payment Processing, Notifications, Chat, and Admin. The AI services layer runs as separate Python FastAPI servers for the Chatbot (Groq LLM + ChromaDB) and CNN breed prediction. The data layer uses PostgreSQL, Redis, ChromaDB, and local file storage. External integrations include Groq LLM API, OpenStreetMap, Firebase Cloud Messaging, and Blockchain (BTC/ETH/USDT).

\subsection{Modular Monolithic Architecture}
All backend modules deploy as a single Golang binary with clear boundaries and interfaces. Benefits include direct function calls for inter-module communication, simplified deployment, centralized logging, atomic transactions, and lower infrastructure costs compared to a microservices approach.

\subsection{Layered Architecture}
The system implements six-tier separation of concerns: Presentation Layer (Flutter mobile app and Next.js admin portal), API Gateway and Routing Layer (Gin HTTP router with JWT, validation, logging, and CORS middleware), Business Logic Layer (Golang modular monolith with all domain modules), AI Services Layer (Python FastAPI servers for CNN and RAG chatbot), Data Storage Layer (PostgreSQL, Redis, ChromaDB, local file system), and External Integrations Layer (Groq LLM, OpenStreetMap, Firebase FCM, blockchain).

\subsubsection{Level 2: Detailed Sub-Processes}

\textbf{Pet Toy Marketplace}

The Level 2 DFD for the Marketplace module decomposes the e-commerce process into six sub-processes: product browsing and search, seller product listing management, shopping cart management, order placement, order management, and product reviews. Buyers interact with all six processes, sellers manage their listings and orders, and the Admin oversees listings and all orders. Six data stores support these processes: Products, Categories, Cart, Orders, Reviews, and Users.

\textbf{Community Hub}

The Level 2 DFD for the Community Hub module decomposes into four sub-processes: Community Forum (posts, comments, likes, groups, hashtags), Lost \& Found Alerts, Pet Adoption Listings, and Events Management (including RSVP). Community Members interact with all four processes and the Admin moderates across all of them. Seven data stores support these processes: Posts, Comments, Groups, Lost \& Found, Adoptions, Events, and Users.

\textbf{Pet Caregiver Services}

The Level 2 DFD for the Pet Caregiver Services module decomposes into six sub-processes: Caregiver Profile Management (profile, services, availability, gallery), Search and Browse Caregivers, Booking Management (create, accept/reject, cancel), Live GPS Tracking (real-time location updates and route history), Service Completion and Reviews (completion report, mutual reviews, incident reporting), and Payment Processing. Pet Owners and Caregivers interact across all processes while the Admin manages profiles, bookings, and incidents. Eight data stores support these processes: Caregiver Profiles, Services, Availability, Bookings, GPS Tracking, Reviews, Payments, and Users.

\textbf{Admin Portal}

The Level 2 DFD for the Admin Portal decomposes into eight sub-processes: Admin Authentication, Dashboard and Analytics, User and Pet Management, Marketplace Moderation, Appointments and Vet Management, Bookings and Caregiver Management, Community Moderation (posts, lost-found, adoptions, events), and Chat Monitoring. The System Administrator interacts with all eight processes through a dedicated Next.js web application. Nine data stores are accessed: Users, Pets, Marketplace, Appointments, Bookings, Community, Vets, Caregivers, and Chats.

\textbf{Payment Integration}

The Level 2 DFD for the Payment Integration module decomposes into five sub-processes: Payment Initiation (triggered from marketplace orders and caregiver bookings), Cryptocurrency Processing (Bitcoin, Ethereum, USDT with QR code generation and blockchain confirmation), Digital Wallet Management (balance top-up, deductions, and transaction history), Transaction Logging and Invoicing (full audit trail with PDF invoice generation), and Refund and Dispute Management (admin-mediated refunds with automatic wallet crediting). Pet Owners initiate payments, Sellers and Caregivers receive funds, and the Admin resolves disputes. Five data stores support these processes: Transactions, Wallets, Orders, Bookings, and Users.

\subsection{Data Storage Strategies}
\begin{itemize}
    \item \textbf{PostgreSQL:} Primary database storing user accounts, pet profiles, health records, marketplace data, and transactions with structured relationships.
    \item \textbf{Redis Caching:} Temporary storage for active user sessions, frequently accessed pet data, and API responses to improve system performance.
    \item \textbf{ChromaDB:} Vector database containing veterinary documents enabling intelligent chatbot responses through semantic search.
\end{itemize}
